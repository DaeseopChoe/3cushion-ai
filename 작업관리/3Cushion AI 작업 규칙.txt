[3Cushion AI — WORKING RULES v1.3]

이 프로젝트는 일반 UI 개발이 아니라
“물리 기반 3쿠션 시스템을 데이터화/검증/시뮬레이션”하는 시스템이다.

모든 작업은 아래 규칙을 반드시 따른다.

---

## 1️⃣ 절대 원칙 (가장 중요)

1. 수식(공식)은 수정하지 않는다
- 시스템 공식은 이미 검증된 값이다
- 문제 발생 시 “수식”이 아니라
  → 입력 / 변환 / 렌더 경로에서 원인을 찾는다

2. 계산은 항상 sys 공간에서만 수행
- 좌표(Fg/Rg)는 표시용이다
- 공식 계산에 좌표를 사용하지 않는다

3. 보정은 항상 Δ_sys로만 적용
- px, grid 등 좌표 보정 금지

4. 추측으로 수정 금지
- 반드시 로그 기반으로 원인 확정 후 수정

---

## 2️⃣ 파이프라인 구조 (핵심 이해)

모든 곡선/라인은 아래 3단계를 따른다:

[계산]
getControlPoint (곡률 생성)

→

[세그먼트 생성]
createCurveSegment (Bezier → points)

→

[렌더]
ImpactLines (polyline)

---

👉 문제는 반드시 이 3곳 중 하나에서 발생한다:

A. control point 문제  
B. segment 생성 문제  
C. 렌더 문제  

---

## 3️⃣ 분석 방법 (반드시 이 순서)

STEP 1: control point 확인  
- mid vs control 비교  
- perp * curvature 실제 적용 여부 확인  

STEP 2: segment points 확인  
- first / mid / last 직선 여부 확인  

STEP 3: 렌더 확인  
- polyline points 확인  
- 필터(isStraightLine 등) 확인  

---

## 4️⃣ 로그 규칙 (필수)

모든 분석은 로그 기반으로 한다.

필수 로그:

[CP]
{ pContact, M, mid, control, curvature }

[SEGMENT OUTPUT]
{ length, first, mid, last }

[DRAW POINTS]
{ length, points }

---

## 5️⃣ 렌더 규칙

1. base line과 curve는 절대 섞지 않는다
- base: 보정 제거 상태
- curve: 보정 포함 결과

2. polyline source는 단일화한다
- cushionPath만 사용

3. 렌더 단계에서 계산 로직 수정 금지

---

## 6️⃣ 데이터/좌표 규칙

- Fg/Rg는 display-space
- sys는 calc-space
- 둘을 혼용하지 않는다

- offset_fg2rg = 2.25 고정 (±0.02 허용)

---

## 7️⃣ 금지 사항

- 수식 변경 금지
- 임의 scaling 금지
- "이렇게 하면 될 것 같아서" 식 수정 금지
- 여러 곳 동시 수정 금지 (단일 원인 원칙)

---

## 8️⃣ 작업 방식

항상 다음 순서로 진행:

1. 로그 추가
2. 원인 확정 (A/B/C)
3. 최소 수정
4. 재검증

---

## 9️⃣ 현재 프로젝트 상태

✔ Base / Effective 완전 분리 완료  
✔ Base Line 렌더 정상  
✔ 잘못된 polyline 제거 완료  

❗ 현재 문제:
→ 곡선(curve)이 직선으로 렌더됨
