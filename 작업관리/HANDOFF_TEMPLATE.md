새 창 이관문서 작성 지침
1. 목적 한 줄
• 왜 옮기는지, 새 창에서 무엇을 이어서 할지 한 문장으로 적는다.
예: “Base Line 세그먼트 파이프라인 이후 남은 버그/UX만 마무리한다.”
2. 저장소·브랜치·커밋 (있으면 필수)
• 저장소 경로, 브랜치 이름, 관련 커밋 해시 또는 커밋 메시지 요약
• “어느 시점까지 반영됐는지”만 알아도 새 GPT가 파일 탐색부터 맞출 수 있다.
3. 이미 한 작업 (사실만)
• 변경된 파일 경로 목록 (App.jsx만 등)
• 동작 요약: 무엇을 고쳤고, 사용자 관점에서 무엇이 달라졌는지 3~5줄
• 의도적으로 안 건드린 것: 예) ImpactLines.jsx 미수정, projection 엔진 미수정
4. 현재 설계·규칙 (새 창이 깨지지 않게)
• Base Line = 보정(slide/draw/spin/curve_ratio) 제거, departure(Sn) 유지 같은 불변 규칙
• 데이터 흐름 한 줄: 예) adminSysNoCorrections → resolvedSlotBaseSysValues → getAnchorsForRendering → CO_path0_base … → cushionPathAttrBase → ImpactLines
5. 해결된 문제 vs 미해결 문제 (분리해서 쓰기)
해결됨
• 증상 → 원인 → 조치 (파일/함수 이름 수준)
아직 / 불명확
• 재현 단계 (클릭 순서, 슬롯, 시스템 종류)
• 기대 vs 실제
• 로그·스크린샷 있으면 “첨부했다”만이라도
6. 새 창에 맡길 “다음 작업” (명령 형태)
• 우선순위 번호 (1이 가장 급함)
• 각 항목마다 완료 조건 한 줄 (예: “Base Line ON 시 노란선이 CO~C2까지 레일 위에만 있다”)
7. 금지·주의 (실수 방지)
• 수정하면 안 되는 파일/영역
• “추측으로 리팩터하지 말 것”, “최소 변경” 같은 사용자 규칙이 있으면 그대로 인용
8. 검증 방법
• 로컬에서 할 명령 (예: npm run build)
• UI에서 확인할 체크리스트 (버튼 ON/OFF, C4까지 표시 등)
9. 열린 질문 (있을 때만)
• 의사결정이 필요한 분기, 문서/코드가 불일치하는 지점
